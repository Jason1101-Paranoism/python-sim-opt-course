# Level 4 週 07 — 實驗與結果呈現（研究版）教師教案

---

## 本週學習目標

1. 學生能設計嚴謹的實驗（控制變因、決定適當的重複次數）
2. 學生能以研究報告標準製作結果圖表（標題、軸標籤、誤差棒、圖說）
3. 學生能以標準學術語氣描述結果（只陳述事實，不解釋原因）
4. 學生能完成 2–3 張關鍵圖表與 1–2 個結果表格

## 各段落執行細節

### 暖身回顧（10 分鐘）

**提問順序**：
1. 「上週方法章初稿，大家大概寫了幾個要素？有沒有卡在哪裡？」（讓每人說 1 句）
2. 「你的評估指標是什麼？單位是什麼？」（快速確認，避免之後結果圖表無單位）
3. 若有學生方法章與 RQ 不對應，簡短點名（「下週 1-on-1 我們來解決這個」），不佔課堂時間

**注意**：若超過半數學生尚未完成方法章初稿，本週概念講解後需預留 10 分鐘給學生先補完方法章的關鍵缺漏，再進行實驗。

### 概念講解（25 分鐘）

**Part 1：什麼是「嚴謹實驗」（10 分鐘）**

引入問題：「你的模擬程式跑了一次，得到一個數字。這個數字可信嗎？」

帶出三個核心概念：

**1. 重複次數（Number of Replications）**

- 隨機模擬每次結果不同，要「跑很多次取平均」才能得到穩定估計
- 怎麼決定跑幾次？
  - 方法一：觀察法——先跑 10 次、50 次、100 次，看平均值是否穩定收斂
  - 方法二：看計算資源——你的程式跑 1000 次要幾秒？可接受的話就跑 1000 次
- Level 4 的基本要求：**每個情境至少 100–500 次**（視計算時間而定）

**2. 控制變因（Control Variables）**

- 每次只能改一個自變數，其他的全部固定
- 違反控制變因的常見錯誤：「我改了 α，同時也換了初始值」——這樣你不知道是哪個造成結果不同
- 口訣：「每次只動一個旋鈕」

**3. 邊界情境（Boundary Cases）**

- 除了中間的參數值，記得也測試極端值（最小、最大）
- 邊界情境往往會出現有趣的行為，也是討論章的好材料

**Part 2：結果呈現的研究標準（15 分鐘）**

一張「研究等級」的圖應有：
- 圖標題（caption）：「Figure 1. [說明圖的內容]」
- 橫軸標籤 + 單位
- 縱軸標籤 + 單位
- 若有多條線：圖例（legend）
- 若有重複實驗：誤差棒（error bar）或陰影區間（confidence interval）
- 字體夠大（至少 12pt），線條夠粗（linewidth ≥ 2）

結果表格應有：
- 表格標題（caption）：「Table 1. [說明表的內容]」
- 每欄有欄位名稱 + 單位
- 數字對齊，保留適當小數位（通常 2–3 位）

### 寫作與圖表示範（30 分鐘）

**從空白程式開始，邊做邊說明。**

**示範情境**：以 SIR 模型為例，展示如何做嚴謹實驗並製作研究等級圖表

```python
import numpy as np
import matplotlib.pyplot as plt
from scipy.integrate import odeint

# SIR 模型（確定性）
def sir_model(y, t, beta, gamma):
    S, I, R = y
    N = S + I + R
    dSdt = -beta * S * I / N
    dIdt = beta * S * I / N - gamma * I
    dRdt = gamma * I
    return [dSdt, dIdt, dRdt]

# 實驗設計：改變 R0（基本傳染數），其他條件固定
N = 10000
I0, R0_init, S0 = 10, 0, N - 10
gamma = 0.1  # 固定不變
t = np.linspace(0, 160, 160)

# 系統性改變 beta（等同改變 R0 = beta/gamma）
beta_values = [0.15, 0.20, 0.25, 0.30]  # 對應 R0 = 1.5, 2.0, 2.5, 3.0
results = {}

for beta in beta_values:
    y0 = [S0, I0, R0_init]
    sol = odeint(sir_model, y0, t, args=(beta, gamma))
    results[beta] = sol

# 製作研究等級圖
fig, ax = plt.subplots(figsize=(8, 5))
for beta in beta_values:
    R0_val = beta / gamma
    peak_infected = results[beta][:, 1].max()
    ax.plot(t, results[beta][:, 1], linewidth=2, label=f'R₀ = {R0_val:.1f}')

ax.set_xlabel('Time (days)', fontsize=12)
ax.set_ylabel('Number of Infected Individuals', fontsize=12)
ax.set_title('Figure 1. Effect of Basic Reproduction Number on Epidemic Curve', fontsize=12)
ax.legend(fontsize=11)
ax.tick_params(labelsize=11)
plt.tight_layout()
plt.savefig('figure1_epidemic_curve.png', dpi=150)
plt.show()
```

**刻意犯錯讓學生找**：故意把 `xlabel` 和 `ylabel` 對調，或忘記 `linewidth`，讓學生指出。

**示範結果表格製作**：

```python
# 製作結果表格
print("Table 1. Summary of Peak Infected Counts by R0")
print(f"{'R0':>6} | {'Peak Infected (n)':>18} | {'Peak Day':>10} | {'Epidemic Duration (days)':>25}")
print("-" * 65)
for beta in beta_values:
    R0_val = beta / gamma
    I_curve = results[beta][:, 1]
    peak_n = int(I_curve.max())
    peak_day = int(t[I_curve.argmax()])
    # 定義疫情結束：I < 1
    duration = int(t[I_curve > 1].max()) if (I_curve > 1).any() else 0
    print(f"{R0_val:>6.1f} | {peak_n:>18d} | {peak_day:>10d} | {duration:>25d}")
```

**示範結果段落寫法（只陳述，不解釋）**：

「如圖 1 所示，隨著基本傳染數（R₀）由 1.5 增加至 3.0，感染人數峰值由 [X] 人增加至 [Y] 人，且峰值出現時間由第 [a] 天提前至第 [b] 天。如表 1 所示，當 R₀ = 3.0 時，疫情持續時間最短，為 [c] 天；當 R₀ = 1.5 時，疫情持續時間最長，為 [d] 天。」

**強調**：這段只說「是什麼」，不說「為什麼」——為什麼留到討論章。

### 學生實作（35 分鐘）

**任務**：學生按自己的研究題目，完成以下任一工作（依進度選擇）：

選項 A（進度正常）：完成第一張關鍵圖（研究等級，有標題、軸標籤、單位）

選項 B（進度超前）：完成 1 個結果表格 + 第一段結果描述文字

選項 C（進度稍落後）：先確認實驗設計（參數範圍、重複次數），並跑通一個基礎情境

**教師巡視重點**：
- 圖表有沒有標題、軸標籤、單位
- 結果文字有沒有混入解釋（「因為 X 所以 Y」應留到討論章）
- 重複次數是否合理（若學生隨機模擬只跑了 10 次，提醒要增加）

---

### 回顧與預告（20 分鐘）

**本週 3 個關鍵點**（白板寫下）：

1. 嚴謹實驗 = 控制變因 + 足夠的重複次數 + 測試邊界情境
2. 研究等級的圖 = 標題 + 軸標籤 + 單位 + 圖例（如有） + 誤差棒（如有）
3. 結果段落只說「是什麼」，解釋「為什麼」留到討論章

**作業說明**（詳見作業單）：完成主要實驗 + 2–3 張關鍵圖 + 1–2 個結果表格 + 結果段落草稿

**預告週 08**：「下週我們進入最需要思考的部分：討論章。同樣一個結果，你要解釋它的意思、把它連結到文獻、說出它的意義。這三個層次是很多學生最卡的地方。」

## 範例文字：好的 vs 不好的結果段落

### 範例情境：最佳化不同服務台數量對顧客等待時間的影響

**不好的版本**：
> 我發現 3 個窗口的時候等待時間最短，這是因為顧客多的時候一個窗口不夠用，所以需要更多。4 個窗口效果也不錯，但感覺浪費資源。

**問題**：
- 「我發現」——口語，應用「結果顯示」「如圖 X 所示」
- 「這是因為...所以...」——在結果章解釋原因是錯誤的，這應該在討論章
- 「感覺浪費資源」——主觀評語，沒有量化依據

**好的版本**：
> 如圖 2 所示，平均等待時間隨服務台數量的增加而單調遞減。當服務台數量由 1 個增加至 3 個時，平均等待時間由 18.4 分鐘降至 3.2 分鐘，降幅達 82.6%。服務台數量由 3 個增加至 4 個時，等待時間進一步降至 1.8 分鐘，但降幅縮小至 43.8%。如表 2 所示，在顧客到達率 λ = 10 人/小時的條件下，3 個服務台的系統利用率（server utilization）為 83.3%，4 個服務台的系統利用率降至 62.5%。

## 常見問題與處理方式

| 學生問題 | 建議回應方式 |
|----------|-------------|
| 「我的模擬跑一次要很久，不能跑 100 次」 | 「先跑 10 次看看，如果結果差很多，說明你需要更多次。可以考慮縮小參數範圍或簡化模型，讓一次模擬更快。」 |
| 「我的結果和我預期的相反，怎麼辦？」 | 「先確認有沒有 bug。如果結果是正確的，這恰好是很有趣的發現——討論章可以解釋為什麼。」 |
| 「圖表要用中文還是英文標籤？」 | 「依你的報告語言決定。如果你最終要投英文競賽，就用英文；若是中文報告，用中文。保持全文一致。」 |
| 「我不知道誤差棒要怎麼算」 | 「用 np.mean 算平均，np.std 算標準差，然後用 ax.errorbar 或 ax.fill_between 加上一個標準差的範圍。」 |
| 「我的結果寫著寫著就開始解釋了，怎麼分辨？」 | 「問自己：這句話是在說『數字是多少』還是在說『為什麼是這個數字』？後者留到討論章。」 |
